Это стандартная утилита в Linux для мониторинга загрузки центрального процессора (ЦП) и активности ввода-вывода (I/O) для блочных устройств и разделов. Она незаменима для поиска "узких мест" в дисковой подсистеме и балансировки нагрузки.

```bah
iostat [опции] [устройство] [интервал [количество]]
```
#### Полезные опции:
- **Выбор данных:** `-c` **(только ЦП)**, `-d` **(только диски)**.
- **Формат вывода:** `-h` **(удобный для чтения)**, `-k` **(в КБ)** `-m` **(в МБ)**, `-t` (с меткой времени)
- **Расширенная статистика:** `-x` **(детальный отчёт)**
- **Фильтрация:** `-p [устройство]` **(статистика по разделу)**, `-z` **(скрыть неактивные диски)**
- **Управление отчётами:** `-y` **(исключить первый отчёт с момента загрузки)**
#### Примеры команд:
```bash
#обновление каждые 5 секунды
iostat 5 

#расширенная статистика каждую секунду (10 раз)
iostat -hx 1 10 

#отчёт только по диску `sda` и его разделам
iostat -p nvme0n1

#мониторинг только устройств в мегабайтах каждые 5 секунд 2 раза
iostat -d -m 5 2

#мониторинг cтатистики каждую секунду (за счёт watch)
watch -n 1 iostat -ht
```

## Основные режимы вывода
### ### Стандартный вывод (`iostat -h`)
- **`%user`**: Время ЦП на выполнение пользовательских приложений
- **`%nice`**: Время ЦП на процессы с изменённым приоритетом `nice`
- **`%system`**: Время ЦП на выполнение кода ядра
- **`%iowait`**: Время простоя ЦП в ожидании завершения I/O операций — ключевой показатель перегрузки дисковой подсистемы
- **`%steal`**: Время ожидания виртуального ЦП, пока гипервизор обслуживает другие виртуальные машины (для сред виртуализации)
- **`%idle`**: Время полного простоя ЦП

- **`tps`**: Общее количество операций ввода-вывода в секунду (Transfers Per Second)
- **`kB_read/s`** и **`kB_wrtn/s`**: Объём данных (в КБ/с), прочитанных и записанных на устройство за секунду
- **`kB_read`** и **`kB_wrtn`**: Общий объём данных (в КБ), прочитанных и записанных с момента загрузки системы
---
### Расширенный вывод (`iostat -hx`)
- **`r/s`** и **`w/s`**: Количество запросов на чтение/запись в секунду (более детально, чем `tps`).
- **`rkB/s`** и **`wkB/s`**: Объём данных в КБ, прочитанных и записанных в секунду.
- **`avgqu-sz`**: Средняя длина очереди запросов к устройству. Высокое значение — признак "узкого места"
- **`await`**: Среднее время ожидания выполнения запроса, включая ожидание в очереди и обработку диском
- **`svctm`**: Среднее время непосредственной обработки запроса диском (рекомендуется использовать `await`)
- **`%util`**: Процент времени, в течение которого устройство было занято обработкой запросов. Значения, близкие к 100%, указывают на предел производительности
---
## На что обратить внимание
Чем больше, тем хуже:

**`%iowait`**  — Процессор ждёт диск
**`avgqu-sz`** — Длина очереди запросов к диску
**`%util`** — Процент времени, когда диск был занят
### Сочетания значений и их расшифровка
#### `%iowait` ↑ (>20) + `%util` ≈ 100% + `await` ↑ (>20ms)
Диск упирается в свой физический предел

| Метрика        | Значение | Метрика           | Значение |
| -------------- | -------- | ----------------- | -------- |
| `%iowait`(cpu) | >20%     | `await` (disk)    | > 65     |
| `%util` (disk) | 100.00%  | `avgqu-sz` (disk) | > 12.5   |

**Что видим:**
- `%iowait` = **35.80** (процессор ждёт диск)    
- `avgqu-sz` = **12.50** (огромная очередь)
- `%util` = **100.00** (диск занят непрерывно)    
- `await` = **65.20 мс** (каждый запрос идёт больше 60 мс)

**Что это значит:** Диск физически не успевает обрабатывать поток запросов. Запросы скапливаются в очереди, процессор простаивает в ожидании данных. Это классический **IO Bottleneck**.  

**Что делать:** Переносить данные на более быстрый носитель (SSD/NVMe) или балансировать нагрузку (например, разнести базу данных и логи на разные физические диски).
#### `await` ↑↑ (>100ms) + `%util` ↓ (<50%) + `r/s`/`w/s` ↓
Медленное устройство (латенция)

| Метрика       | Значение | Метрика            | Значение |
| ------------- | -------- | ------------------ | -------- |
| `await`(disk) | >100 мс  | `avgqu-sz`(disk)   | > 1      |
| `%util`(disk) | <50%     | `r/s`, `w/s`(disk) | 20       |

**Что видим:**
- `%util` = 50 (диск занят меньше половины времени)
- `r/s` + `w/s` = 20 (операций очень мало)
- `await` = **250.00 мс** (катастрофическая задержка выполнения запроса)
- `avgqu-sz` = **1.80** (очередь растёт из-за того, что диск долго "думает" над каждым запросом)

**Что это значит:** Диск вроде бы свободен (`%util` низкий), но каждое обращение к нему выполняется катастрофически долго.  

**Вероятные причины:**
1. **Ошибки на диске / бэд-блоки.** Система пытается перечитывать сектора (проверить `dmesg | grep -i error`).
2. **Внешнее хранилище (SAN/NAS).** Проблемы с сетью, перегрузка контроллера массива или синхронная запись на медленный NFS.
3. **HDD с парковкой головок.** Диск засыпает, и первый запрос будит его с огромной задержкой.
#### `r/s` + `w/s` ↑↑ (>1000) + `rkB/s`/`wkB/s` ↓ + `%util` ≈ 100%
Шторм мелких запросов (IOPS bound)

| Метрика             | Значение        | Метрика                 | Значение          |
| ------------------- | --------------- | ----------------------- | ----------------- |
| `r/s` + `w/s`(disk) | Высокие (>1000) | `rkB/s` / `wkB/s`(disk) | Небольшие         |
| `%util`(disk)       | Высокий / 100%  | `await`(disk)           | Средний (1-10 мс) |
	**Что видим:**
- `r/s` + `w/s` = **1250** (тысячи операций в секунду).
- `rkB/s` + `wkB/s` = **5000 kB/s** ≈ 5 MB/s (объём данных смешной, а диск нагружен).
- `%util` = **99.50** (диск забит операциями).
- `await` = **7.50 мс** (в пределах нормы для HDD при случайном доступе).

**Что это значит:** Диск (особенно HDD) умирает не от объёма данных, а от количества операций в секунду (чтение файловых атрибутов, индексов БД).  

**Пример:** 1000 операций случайного чтения по 4Кб дадут всего ~4 МБ/с трафика, но полностью забьют механику HDD. На SSD такого эффекта почти нет.
#### `rMB/s` или `wMB/s` ↑↑ (>50 MB/s) + `avgqu-sz` > 1 + `await` в норме
Нормальная последовательная нагрузка

| Метрика       | Значение                     | Метрика    | Значение              |
| ------------- | ---------------------------- | ---------- | --------------------- |
| `rkB/s`(disk) | Высокое (>50 МБ/c)           | `avgqu-sz` | **> 1**               |
| `await`       | Нормальный для HDD (5-20 мс) | `%iowait`  | **Может быть низким** |
**Что видим:**
- `wMB/s` = **95.00 MB/s** (поток записи на пределе пропускной способности HDD).
- `avgqu-sz` = **3.50** (очередь есть, но движется планомерно).
- `await` = **80.00 мс** — **это норма для HDD при последовательной записи с большой очередью**. Узкое место не в механике головок, а в скорости вращения пластин.

**Что это значит:** Кто-то пишет большой файл (бэкап, лог, видео). Очередь есть, но она движется быстро. Это **не проблема** производительности приложения, если оно не чувствительно к задержкам. Высокий `%iowait` при последовательной записи на HDD - тоже норма.
#### `%iowait` ≈ 0% + `%util` ↓ (<20%) + `%user`/`%system` ↑ (близко к 100%)
Проблема не в диске (CPU bound)

| Метрика             | Значение       |
| ------------------- | -------------- |
| `%iowait`           | 0-5%           |
| `%util`             | <20%           |
| `%user` / `%system` | Высокие / 100% |
**Что видим:**
- `%iowait` = **0.00** (диск не заставляет процессор ждать)
- `%user` + `%system` = **100%** (процессор занят полностью вычислениями или кодом ядра)
- `%util` = **5.00** (диск почти простаивает)
- **Вердикт:** Диск не виноват. Искать проблему нужно через `top` или `perf`

**Что это значит:** Пользователь жалуется на тормоза, но `iostat` показывает, что диск спит. Тормозит **процессор** или **оперативная память** (проверять `top` / `vmstat`).
