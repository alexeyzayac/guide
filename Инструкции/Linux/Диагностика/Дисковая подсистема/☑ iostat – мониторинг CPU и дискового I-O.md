# Команда `iostat` – мониторинг CPU и дискового I/O

**`iostat`** (из пакета `sysstat`) – стандартная утилита Linux для сбора и отображения статистики загрузки процессора и активности блочных устройств (дисков, SSD, NVMe). Незаменима для поиска «узких мест» в дисковой подсистеме, анализа очередей и балансировки нагрузки.
## ⚙️ Популярные команды
```bash
iostat -x 2               # расширенная статистика, обновление каждые 2 секунды
iostat -d -m 5 3          # только диски в мегабайтах, 3 отчёта с шагом 5 с
iostat -p nvme0n1 1       # по разделу nvme0n1 (p1, p2...) и сам диск
watch -n 1 iostat -h      # мониторинг в реальном времени через watch
iostat -hx 1 10 | grep -v "^\$"  # убрать пустые строки
```
## 📖 Полезные опции
|Опция|Длинная форма|Назначение|
|---|---|---|
|**Выбор данных**|||
|`-c`|–|Только статистика **CPU**|
|`-d`|–|Только статистика **дисков**|
|`-x`|`--extended`|Расширенная статистика (avgqu-sz, await, %util и др.)|
|`-N`|`--lvm`|Показывать логические тома LVM вместо отдельных дисков|
|`-j`|`--json`|Вывод в формате JSON (удобно для скриптов)|
|**Единицы измерения**|||
|`-h`|`--human-readable`|Читаемые единицы (К, М, Г)|
|`-k`|–|Вывод в килобайтах (по умолчанию)|
|`-m`|–|Вывод в мегабайтах|
|`-t`|–|Добавить метку времени|
|**Управление отчётами**|||
|`-y`|`--no-first`|Пропустить первый (накопительный с момента загрузки) отчёт|
|`-z`|–|Скрывать неактивные устройства (`tps = 0`)|
|`-p [устр]`|–|Показывать статистику по разделам (например `-p sda`)|
|`-o`|–|Вывод в старом формате (без дополнительных полей)|
|`-s`|–|Не выводить сообщения о создании скользящих окон (для скриптов)|
|`-V`|–|Версия утилиты|

> 💡 `-x` включает большинство важных метрик. `-y` полезен при длительном мониторинге, чтобы убрать первый «усреднённый» отчёт.

## ⚙️ Быстрый старт
```bash
# 1. Мониторинг диска sda каждые 5 секунд, 3 отчёта, человеко-читаемый
iostat -d -h sda 5 3

# 2. Расширенная статистика всех дисков, обновление каждую секунду (10 раз)
iostat -hx 1 10

# 3. Только CPU, скрыть первый отчёт, интервал 2 секунды
iostat -c -y 2

# 4. Статистика по всем разделам /dev/nvme0n1
iostat -p nvme0n1 2

# 5. LVM-тома и физические диски под ними
iostat -N 2

# 6. JSON для последующей обработки (например, в скрипте)
iostat -x -j JSON 1 1 | jq '.sysstat.hosts[0].statistics[0]'

# 7. Комбинированный мониторинг: CPU и диски в расширенном виде
iostat -c -x 2
```
### Секция CPU (режимы `-c` или без опций)

|Поле|Значение|Что значит|
|---|---|---|
|`%user`|Время на пользовательские процессы (приложения)|Норма: <70%, высокое – приложения грузят CPU|
|`%nice`|То же для процессов с изменённым приоритетом `nice`|Обычно 0, если не используется приоритетизация|
|`%system`|Время на задачи ядра (системные вызовы, драйверы)|Высокое – возможно, проблема с драйверами или частые контекстные переключения|
|`%iowait`|**Процессор ждёт завершения I/O**|>20% – диск не успевает, «узкое горлышко» I/O|
|`%steal`|Время ожидания виртуального CPU (гипервизор занят)|>5% – перегружен физический хост|
|`%idle`|Полное безделье CPU (ничего не делает)|Низкое (%idle ~0) при высокой нагрузке – норма|

### Секция дисков (режим `-d` или по умолчанию)

|Поле|Значение|Замечание|
|---|---|---|
|`tps`|Передач в секунду (чтение+запись)|Один запрос может быть разного размера|
|`kB_read/s`|Килобайт чтения в секунду|Скорость чтения данных|
|`kB_wrtn/s`|Килобайт записи в секунду|Скорость записи|
|`kB_read`|Всего прочитано (с момента загрузки)|Накопленное значение (в первом отчёте)|
|`kB_wrtn`|Всего записано (с момента загрузки)|–|

### Расширенная статистика (опция `-x`)

|Поле|Смысл|Пороговое значение|
|---|---|---|
|`rrqm/s`|Слитых запросов чтения (merge)|Высокое – признак последовательной нагрузки|
|`wrqm/s`|Слитых запросов записи|–|
|`r/s`|Чтений в секунду (IOPS чтения)|Для HDD > 100–200 – плохо|
|`w/s`|Записей в секунду (IOPS записи)|–|
|`rkB/s`|КБ/с чтения|–|
|`wkB/s`|КБ/с записи|–|
|`avgrq-sz`|Средний размер запроса (в секторах по 512 байт)|Малые значения (< 16) – случайный I/O|
|`avgqu-sz`|**Средняя длина очереди**|>1 – нагрузка выше пропускной способности|
|`await`|**Среднее время выполнения запроса (мс)**|>20 мс для HDD – перегрузка; >100 мс – проблемы|
|`r_await`|Время чтения (мс)|–|
|`w_await`|Время записи (мс)|–|
|`svctm`|**Устаревшее** (среднее время обслуживания)|Не использовать, всегда смотрите `await`|
|`%util`|**Процент времени, когда устройство было занято**|~100% – диск работает на пределе|
## 🔍 Типовые паттерны и расшифровка
Чем больше, тем хуже:

**`%iowait`**  — Процессор ждёт диск
**`avgqu-sz`** — Длина очереди запросов к диску
**`%util`** — Процент времени, когда диск был занят
### Сочетания значений и их расшифровка
#### `%iowait` ↑ (>20) + `%util` ≈ 100% + `await` ↑ (>20ms)
Диск упирается в свой физический предел

| Метрика        | Значение | Метрика           | Значение |
| -------------- | -------- | ----------------- | -------- |
| `%iowait`(cpu) | >20%     | `await` (disk)    | > 65     |
| `%util` (disk) | 100.00%  | `avgqu-sz` (disk) | > 12.5   |

**Что видим:**
- `%iowait` = **35.80** (процессор ждёт диск)    
- `avgqu-sz` = **12.50** (огромная очередь)
- `%util` = **100.00** (диск занят непрерывно)    
- `await` = **65.20 мс** (каждый запрос идёт больше 60 мс)

**Что это значит:** Диск физически не успевает обрабатывать поток запросов. Запросы скапливаются в очереди, процессор простаивает в ожидании данных. Это классический **IO Bottleneck**.  

**Что делать:** Переносить данные на более быстрый носитель (SSD/NVMe) или балансировать нагрузку (например, разнести базу данных и логи на разные физические диски).

---
#### `await` ↑↑ (>100ms) + `%util` ↓ (<50%) + `r/s`/`w/s` ↓
Медленное устройство (латенция)

| Метрика       | Значение | Метрика            | Значение |
| ------------- | -------- | ------------------ | -------- |
| `await`(disk) | >100 мс  | `avgqu-sz`(disk)   | > 1      |
| `%util`(disk) | <50%     | `r/s`, `w/s`(disk) | 20       |

**Что видим:**
- `%util` = 50 (диск занят меньше половины времени)
- `r/s` + `w/s` = 20 (операций очень мало)
- `await` = **250.00 мс** (катастрофическая задержка выполнения запроса)
- `avgqu-sz` = **1.80** (очередь растёт из-за того, что диск долго "думает" над каждым запросом)

**Что это значит:** Диск вроде бы свободен (`%util` низкий), но каждое обращение к нему выполняется катастрофически долго.  

**Вероятные причины:**
1. **Ошибки на диске / бэд-блоки.** Система пытается перечитывать сектора (проверить `dmesg | grep -i error`).
2. **Внешнее хранилище (SAN/NAS).** Проблемы с сетью, перегрузка контроллера массива или синхронная запись на медленный NFS.
3. **HDD с парковкой головок.** Диск засыпает, и первый запрос будит его с огромной задержкой.

---
#### `r/s` + `w/s` ↑↑ (>1000) + `rkB/s`/`wkB/s` ↓ + `%util` ≈ 100%
Шторм мелких запросов (IOPS bound)

| Метрика             | Значение        | Метрика                 | Значение          |
| ------------------- | --------------- | ----------------------- | ----------------- |
| `r/s` + `w/s`(disk) | Высокие (>1000) | `rkB/s` / `wkB/s`(disk) | Небольшие         |
| `%util`(disk)       | Высокий / 100%  | `await`(disk)           | Средний (1-10 мс) |
	**Что видим:**
- `r/s` + `w/s` = **1250** (тысячи операций в секунду).
- `rkB/s` + `wkB/s` = **5000 kB/s** ≈ 5 MB/s (объём данных смешной, а диск нагружен).
- `%util` = **99.50** (диск забит операциями).
- `await` = **7.50 мс** (в пределах нормы для HDD при случайном доступе).

**Что это значит:** Диск (особенно HDD) умирает не от объёма данных, а от количества операций в секунду (чтение файловых атрибутов, индексов БД).  

**Пример:** 1000 операций случайного чтения по 4Кб дадут всего ~4 МБ/с трафика, но полностью забьют механику HDD. На SSD такого эффекта почти нет.

---
#### `rMB/s` или `wMB/s` ↑↑ (>50 MB/s) + `avgqu-sz` > 1 + `await` в норме
Нормальная последовательная нагрузка

| Метрика       | Значение                     | Метрика    | Значение              |
| ------------- | ---------------------------- | ---------- | --------------------- |
| `rkB/s`(disk) | Высокое (>50 МБ/c)           | `avgqu-sz` | **> 1**               |
| `await`       | Нормальный для HDD (5-20 мс) | `%iowait`  | **Может быть низким** |
**Что видим:**
- `wMB/s` = **95.00 MB/s** (поток записи на пределе пропускной способности HDD).
- `avgqu-sz` = **3.50** (очередь есть, но движется планомерно).
- `await` = **80.00 мс** — **это норма для HDD при последовательной записи с большой очередью**. Узкое место не в механике головок, а в скорости вращения пластин.

**Что это значит:** Кто-то пишет большой файл (бэкап, лог, видео). Очередь есть, но она движется быстро. Это **не проблема** производительности приложения, если оно не чувствительно к задержкам. Высокий `%iowait` при последовательной записи на HDD - тоже норма.

---
#### `%iowait` ≈ 0% + `%util` ↓ (<20%) + `%user`/`%system` ↑ (близко к 100%)
Проблема не в диске (CPU bound)

| Метрика             | Значение       |
| ------------------- | -------------- |
| `%iowait`           | 0-5%           |
| `%util`             | <20%           |
| `%user` / `%system` | Высокие / 100% |
**Что видим:**
- `%iowait` = **0.00** (диск не заставляет процессор ждать)
- `%user` + `%system` = **100%** (процессор занят полностью вычислениями или кодом ядра)
- `%util` = **5.00** (диск почти простаивает)
- **Вердикт:** Диск не виноват. Искать проблему нужно через `top` или `perf`

**Что это значит:** Пользователь жалуется на тормоза, но `iostat` показывает, что диск спит. Тормозит **процессор** или **оперативная память** (проверять `top` / `vmstat`).
